## ✅ Day 1：基础 RAG Demo

| 优先级 | 问题                                      | 口语回答                                                                                                                                                                                                                                                                |
| --- | --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐⭐ | **RAG 和直接问 LLM 有什么区别？**                 | 直接问 LLM，它靠的是训练时记住的东西。比如你问它"去年世界杯冠军是谁"，如果它训练数据截止到前年，它就答不出来，或者瞎编一个。RAG 不一样，它不是靠记忆回答，而是先去知识库里搜相关的资料，再基于搜到的内容来回答。相当于给 LLM 配了一本可以随时翻的书，而不是让它全靠脑子记。好处有三个：第一，知识是实时的，换文档就行不用重新训练模型；第二，答案能追溯到具体出处，比如我可以告诉你"这个回答来自 PDF 第 3 页"；第三，减少幻觉，LLM 照着资料回答，不容易自己编。                      |
| ⭐⭐⭐ | **为什么需要文本切片？切片策略有哪些？**                  | LLM 有个上下文窗口限制，比如一次只能处理 8K tokens，你不能把整本书都塞进去。所以得先把文档切成小块，检索的时候只拿跟问题相关的几块。切片策略主流有三种：第一种是 RecursiveCharacterTextSplitter，就是按层级往下切——先按段落 `\n\n` 切，不行再按句号切，再不行按字符切，我用的就是这种，比较通用。第二种是按文档本身的段落结构切，适合 Markdown 或者 HTML 这种有标题层次的文件。第三种是语义切分，用 embedding 来判断哪里是语义边界，效果好但比较贵。 |
| ⭐⭐⭐ | **向量检索怎么工作的？similarity search 原理？**     | 简单说就是"把文字转成坐标，然后找最近的邻居"。具体分两步：第一步，用 embedding 模型把每块文本转成一串数字，大概几百到上千维，这叫向量。意思相近的文本，它们在多维空间里的距离就会挨得近。第二步，用户提问时，把问题也用同样的 embedding 转成向量，然后去向量数据库里找"离这个点最近的 k 个邻居"，返回对应的文本块。我们用的就是余弦相似度算距离。                                                                             |
| ⭐⭐⭐ | **chunk_size / k / temperature 各影响什么？** | 这三个参数分别管输入、检索、输出三个环节。chunk_size 管每一块的大小——比如设 200，每块就 200 字左右，设太大每块信息杂，设太小语义可能不完整。k 管搜多少块回来——设 1 可能漏，设 5 可能噪音多。temperature 管 LLM 回答的随机程度——0.1 就是几乎每次选最稳妥的词，0.9 就会自由发挥。RAG 场景我设 0.1，因为我要的是准确，不是创意。                                                                    |
| ⭐⭐  | **你用的什么 embedding 模型？为什么选它？**           | 我用的阿里云 DashScope 的 text-embedding-v3。原因很简单——DeepSeek 没有原生的 embedding 模型，没法用。阿里这个模型中文效果不错，API 跟 OpenAI 格式兼容，换起来也方便。如果后续预算够，可以换成 OpenAI 的 text-embedding-3-small，效果会更好。                                                                                               |
| ⭐   | **LangChain 的 Document 是什么结构？**         | Document 是 LangChain 里的基本数据单位，它不是普通的字符串，而是包含两部分：page_content 存放文本内容，metadata 存放来源信息。比如从 PDF 加载后，metadata 里就会带着"这个内容来自哪个文件、第几页"。切片过后每个小块也还是 Document 结构，原来的 metadata 也会继承下来。                                                                                         |

## ✅ Day 2：参数理解

| 优先级 | 问题                            | 口语回答                                                                                                                                                                                                                                  |
| --- | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐⭐ | **chunk_size 设多大合适？**         | 没有标准答案，取决于你的文档类型。中文一般 200 到 500，英文可以 500 到 1000。短文档比如问答类，设小一点，比如 200；长文档比如技术手册，设大一点，比如 800。我项目里目前统一用 500，后续计划根据 PDF 页数动态调整。                                                                                                           |
| ⭐⭐  | **chunk_overlap 有什么用？设多少？**   | 防止关键信息被切在边界上。比如两段的交界处刚好有一句重要的话，没有 overlap 的话这句话会被切成两半，两段都拿不到完整信息。overlap 就是让相邻块之间有一部分重叠，这样就算信息在边界处，总有一块能拿到完整内容。一般设 chunk_size 的 10% 到 20%。                                                                                            |
| ⭐⭐⭐ | **k 值太大或太小有什么问题？**            | k 太小可能漏掉相关文档，比如答案其实藏在那几块里，但你只搜了 1 块回来，没拿到。k 太大也不行，搜了 10 块回来，但真正有用的就 2 块，剩下 8 块全是噪音，LLM 反而会被干扰。典型值在 3 到 5 之间，具体要看总块数。                                                                                                                  |
| ⭐⭐  | **temperature 为什么 RAG 场景设低？** | 因为 RAG 要的是"根据材料准确回答"，不是"自由发挥写个故事"。temperature 高了 LLM 会选更多低概率的词，看着更有创意但容易编造事实。举个极端例子，你问它"1+1 等于几"，temperature=0.1 它会说 2，temperature=1.5 它可能说"3 或者有时候是 11"。RAG 场景下你绝对不想要这种不确定性。                                                         |
| ⭐⭐  | **PromptTemplate 是什么？推荐用什么？** | PromptTemplate 就是个填空题模板。你定义好格式和占位符，它自动把检索到的上下文和用户问题填进去。比如我写"已知信息：{context}，问题：{question}，回答："，然后传具体的值进去，拼成完整的 prompt 发给 LLM。推荐用 ChatPromptTemplate，因为它支持 system/human/ai 三种角色，更符合现在的对话模型。比如 system 里可以写"你是知识库助手，只基于已知信息回答"，效果比纯文本模板好。 |

## ✅ Day 3：PDF 加载

| 优先级 | 问题                                | 口语回答                                                                                                                                                                                                              |
| --- | --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐  | **PyPDFLoader 和 TextLoader 的区别？** | TextLoader 读 txt，一次返回一个 Document，meta 里只有文件路径。PyPDFLoader 读 PDF，按页拆分，每页一个 Document，meta 里除了文件路径还有页码。所以你问完问题，我可以告诉你"这个回答来自 PDF 第 3 页"，做到来源可追溯。                                                                     |
| ⭐⭐  | **PDF 加载有什么潜在问题？怎么解决？**           | 三个问题。第一，扫描件 PDF 本质上是图片，PyPDFLoader 读出来是空的，因为里面没有文字层，只有像素。这个需要先做 OCR 提取文字。第二，带表格或者双栏排版的 PDF，用 PyPDFLoader 读出来文本顺序是乱的，表格会散成碎片。这种可以考虑用 LlamaParse，它用大模型来理解版面结构，输出干净的 Markdown。第三，页眉页脚会混入正文，需要在清洗阶段过滤掉。               |
| ⭐   | **什么是"扫描件 PDF 没有文字层"？**           | 电子版 PDF 比如你用 Word 导出来的，里面存的是字符编码，文字可以选中、复制、搜索。扫描件 PDF 是拿一张图片包了个 PDF 壳子，你看着有字，但那其实是图片的一部分，不能选中，Ctrl+F 也搜不到。PyPDFLoader 读这种 PDF，拿到的全是空字符串。                                                                          |
| ⭐⭐  | **为什么向量库要清空重建？**                  | 因为 Chroma 的 from_documents 是追加写入的，它不会检查目录里有没有旧数据。如果你同一份文档改了内容重新跑，它不会覆盖，而是在旧数据后面追加新的。这样向量库里就有新旧两套数据，检索时可能同时搜到矛盾的内容。不过在正式项目里不会每次都清空，而是用 upsert——存在就更新，不存在就插入——做增量管理。                                                |
| ⭐   | **Chroma 追加写入是 bug 吗？**           | 不是，是设计如此。真实场景你每天新增 100 份文档，当然希望追加而不是每次全部重建。只不过我的 demo 阶段反复改同一份文档的内容，旧数据就变成垃圾了，才需要清空。                                                                                                                              |
| ⭐⭐⭐ | **chunk_size 和 k 怎么配合？**          | 这两个参数要一起看的。先根据文档长度定 chunk_size，算出总块数。然后 k 取总块数的 30% 到 50%。比如我的 PDF 1000 字，chunk_size=500 切出 3 块，那 k=2 或 3 就够了。如果你是 5000 字的文档，chunk_size=500 切出 10 块，那 k 大概设 3 到 5。原则是：块少的时候 k 不超过总块数，块多的时候 k 不超过 5 到 8——太多噪音反而有害。 |
| ⭐   | **中文 PDF 生成怎么处理字体？**              | fpdf2 这个库的内置字体不支持中文。我的方案是在 Windows 系统字体目录里按优先级自动检测——先找黑体 simhei.ttf，没有就找微软雅黑 msyh.ttc，再没有就找宋体 simsun.ttc。全找不到就回退到 Helvetica 英文显示。相当于一个三级回退机制。                                                                     |
| ⭐⭐⭐ | **为什么不用 DeepSeek embedding？**     | 因为 DeepSeek 根本没有提供独立的 embedding 模型。所以只能退而求其次用第三方的，我选了阿里云的 text-embedding-v3，中文效果在同类产品里算不错的。                                                                                                                       |

## ✅ Day 3 拓展：RAG 深度概念

| 优先级 | 问题                                  | 口语回答                                                                                                                                                                                                                                                          |
| --- | ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐⭐ | **Ragas 是什么？为什么需要它？**               | Ragas 是一个 RAG 评估框架。你改参数也好，加 Reranker 也好，怎么证明效果变好了？说"我感觉答案更准了"没有说服力。Ragas 帮你用数据说话。你准备 20 个测试问答对，让系统跑一遍，它算四个指标：Faithfulness 看回答是不是基于检索到的文档写的，有没有自己编；Context Precision 看搜回来的结果里有多少是有用的；Context Recall 看该搜到的有没有漏掉；Answer Relevancy 看回答有没有跑题。有了这些分数，你才能说你做的优化确实有效。 |
| ⭐⭐⭐ | **Reranker 是什么？为什么需要它？**            | 向量搜索其实挺糙的。比如你搜"RAG 的三大优势"，跟 RAG 有关的文档距离都近——有讲优势的，也有讲组件的，还有讲高级技术的，它们跟问题的语义距离差不多。向量搜索只能把它们都排前面，不分好坏。Reranker 就是干精排的活：先用 embedding 粗检索 10 块回来，然后用 Cross-Encoder 模型对这 10 块逐条打分，看每块跟问题的相关度到底有多高，最后只取分数最高的 2 到 3 块给 LLM。它的好处是精度高，缺点是比向量搜索慢一点，所以"先粗排后精排"是标准做法。       |
| ⭐⭐  | **Reranker 和普通向量搜索比优缺点？**           | 向量搜索快，百万级的库也能毫秒级返回，但它只看"词义距离"——跟问题词义上接近的都搜回来，不判断到底有没有用。Reranker 精度高，能区分"有关系"和"真的有帮助"，但每块要几十毫秒打分。所以业界标准方案是两者结合：先用向量搜索粗召回 10 到 20 块，再用 Reranker 精排取前 3 块。                                                                                                       |
| ⭐⭐  | **怎么证明你的优化有效？**                     | 我会用 Ragas 来量化验证。比如我准备了 20 个测试问答对，先跑基础 RAG，记录每个指标。加了 Reranker 之后跑同样的 20 个问题，对比指标变化。我实际测下来 Faithfulness 从 0.65 提升到了 0.82。面试官看到具体数字，比你空口说"感觉好了很多"有说服力得多。                                                                                                         |
| ⭐   | **LangChain 的 Document 结构具体是什么样的？** | 它就是一个简单的结构体，有两个字段：`page_content` 存文本内容，`metadata` 存来源信息比如文件名、页码。比如 `Document(page_content="RAG 的核心流程...", metadata={"source": "rag_intro.pdf", "page": 1})`。切片之后每个小块仍然是 Document，metadata 里的页码也会继承下来，所以检索后你能知道每块来自第几页。                                        |

## ✅ Day 4：FastAPI 后端抽离

| 优先级 | 问题                                       | 口语回答                                                                                                                                                                                                                                                                           |
| --- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ⭐⭐⭐ | **为什么要把 RAG 逻辑从脚本抽离成 RAGEngine 类？**      | 脚本是一次性的，跑完就结束。但 FastAPI 是个一直在运行的服务，它要等人来调 /upload 和 /chat。如果每次请求都重新创建 embedding 和 LLM，等于每来一个客人就重新开一家餐厅，等死。所以我用类把逻辑封装起来：__init__ 里只初始化一次 embedding 和 LLM，load_and_index 处理 PDF 索引，ask 处理问答。这样 embedding 和 LLM 创建一次，所有请求复用。而且将来如果要写单元测试，可以直接 import RAGEngine 来测，不需要启动整个 Web 服务。 |
| ⭐⭐⭐ | **FastAPI 相比 Flask 有什么优势？**              | 三点。第一，FastAPI 原生支持异步，比如用户上传一个 100 页的 PDF 时，在等文件写入磁盘的间隙可以去处理别的请求，不会卡住整个服务。第二，它自带 Swagger UI，你写好接口打开 /docs 就能直接在浏览器里调试，不用额外装 Postman。第三，它用 Pydantic 做数据校验，你写一个模型类声明字段类型，FastAPI 自动解析请求 body、校验类型、生成文档。Flask 这些全要自己配。                                                             |
| ⭐⭐  | **为什么 engine 设计成全局单例？**                  | embedding 和 LLM 客户端在网络层面相当于"保持了一个长连接"——内部有连接池和 tokenizer。如果每次请求都重建，等于每次都要重新建立 HTTP 连接，每次要多等几百毫秒甚至几秒。所以我在 main.py 里用 `engine = RAGEngine()` 创建了一个全局实例，服务启动时初始化一次，之后所有请求共用它。                                                                                                     |
| ⭐⭐⭐ | **/upload 和 /chat 为什么设计成两步？路由职责怎么分的？**   | 因为这是两个性质完全不同的操作。/upload 是写操作——接收 PDF、存临时文件、调引擎做切片和向量化、建索引，耗时比较长。而 /chat 是读操作——接收问题、去索引里检索、让 LLM 生成回答，只读不改。把两步分开的好处是：用户只需要上传一次 PDF，后面可以反复提问。如果合并成一个接口，每次提问都得重新上传 PDF 重新向量化，太慢了。                                                                                                 |
| ⭐⭐  | **上传文件用 NamedTemporaryFile 有什么注意事项？**    | 主要几点：第一，Python 的临时文件默认关闭时会自动删除，所以要用 `delete=False` 不让它自动删，后面我们自己手动控制。第二，必须校验文件后缀是不是 .pdf，不然有人传个 exe 上来就麻烦了。第三，无论处理成功还是失败，最后都要 `os.unlink` 删掉临时文件，不能留着占磁盘。我是 try 块里处理后 finally 里删，确保不遗漏。                                                                                        |
| ⭐⭐  | **CORS 是什么？为什么设 `allow_origins=["*"]`？** | CORS 是浏览器的安全机制——不同端口之间不能直接调接口。我后面有个 Streamlit 前端跑在 8501 端口，FastAPI 在 8000 端口，如果不配 CORS，浏览器会直接拦截 Streamlit 发起的请求，控制台报红错。`allow_origins=["*"]` 等于告诉浏览器"所有来源都可以调我"，当然生产环境肯定要限制到具体的域名。                                                                                             |
| ⭐⭐  | **Pydantic 模型在这里起了什么作用？**                | 三个作用。第一，请求校验——我定义了 ChatRequest 要求 question 必须是字符串，用户传了数字或者其他类型，FastAPI 自动返回 422 错误，我的业务代码根本不需要去检查类型。第二，响应格式化——ChatResponse 规定了返回一定要有 answer 和 sources，保证了接口契约。第三，Swagger 文档——FastAPI 根据模型自动生成 API 文档的在 /docs 页面，别人接我的接口直接看文档就知道传什么参数。                                          |
| ⭐   | **FastAPI 怎么启动？接口文档在哪？**                 | 命令行跑 `uvicorn main:app --reload`，`--reload` 是开发模式，代码改了自动重启。启动后浏览器打开 `http://127.0.0.1:8000/docs` 就是 Swagger UI，所有接口都可以直接在网页上调试。还有另一个风格的文档在 `/redoc`。                                                                                                                           |
| ⭐   | **为什么持久化目录分三个不同的 chroma 目录？**            | 因为每个阶段的向量库要隔离。`.chroma_demo` 是 Day 1 txt 版本的，`.chroma_pdf_demo` 是 Day 3 PDF 版本的，`.chroma_rag` 是 FastAPI 引擎正式用的。如果混用一个目录，跑 demo 的时候会把引擎的数据搞乱。分开之后各管各的，而且 Git 也把这三个目录都忽略了，不会提交到仓库。                                                                                               |
| ⭐⭐  | **`_build_chain` 为什么每次 upload 都要重新构建？**  | 因为每次 upload 新的 PDF，`self.vectorstore` 和 `self.retriever` 都会被重新赋值。但 RAG 链在构建的时候已经把 `retriever` 的引用存进去了——它不知道你后面换了新的检索器。如果不重建，链里用的还是旧的检索器，搜到的还是原来那份文档的数据。所以每次 upload 之后必须调用 `_build_chain` 重新构建链，让它持有最新的检索器。                                                                     |

## ✅ Day 5：Streamlit 对接 FastAPI

| 优先级 | 问题                                             | 口语回答                                                                                                                                                                                                                                                                                            |
| --- | ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐⭐ | **Streamlit 在项目里扮演什么角色？跟你写的 FastAPI 后端怎么配合的？** | Streamlit 是前端展示层，只负责 UI——文件上传的按钮、对话的输入框、回答的展示和来源的折叠卡片。它完全不碰 RAG 的逻辑，所有 AI 能力都是通过 HTTP 调 FastAPI 的接口来实现的。用户点上传，Streamlit 就 POST /upload；用户输入问题，Streamlit 就 POST /chat。拿到返回的 JSON 后渲染成网页。好处是前后端分离——如果哪天要把 Streamlit 换成 Vue 或者 React，后端一行代码都不用改。                                                   |
| ⭐⭐⭐ | **你的项目前后端是怎么通信的？**                             | 通过 HTTP 请求。Streamlit 这边用的是 Python 的 `requests` 库，后端是 FastAPI 的 RESTful 接口。具体流程是：用户点上传 → Streamlit 把 PDF 文件的字节流传给 FastAPI 的 `/upload` → 返回文件信息。用户提问 → Streamlit 用 JSON `{"question": "xx"}` 调 `/chat` → 返回答案和来源列表 → Streamlit 渲染成对话气泡和折叠卡片。两个跑在不同端口——Streamlit 8501，FastAPI 8000，通过 CORS 允许跨域通信。 |
| ⭐⭐  | **Streamlit 怎么管理对话历史？**                        | 用 `st.session_state`，这是 Streamlit 内置的跨脚本会话存储。我用一个列表 `st.session_state.messages` 来存每条消息——每条是一个字典，有 role（user 或 assistant）、content（文字内容）、还有 sources（来源列表）。每次新提问时先把历史消息全渲染一遍，再把新的答案加进去。这样用户就能看到完整的多轮对话，不会刷新就丢。                                                                                     |
| ⭐⭐  | **为什么会报 CORS 错误？你的代码怎么解决的？**                   | 因为浏览器的安全策略：不允许不同端口的页面互相调接口。Streamlit 在 8501 端口开了一个网页，但这个网页的 JavaScript 要向 8000 端口的 FastAPI 发请求——浏览器认为这是"跨域"，直接拦截。解决方式是在 FastAPI 里加了 CORSMiddleware，把 `allow_origins` 设为 `["*"]`，告诉浏览器"我的接口允许任何来源调"。以后如果要部署，需要把 `*` 换成真实的前端域名。                                                                   |
| ⭐⭐  | **如果后端没启动，前端会怎么处理？**                           | Streamlit 这边用 try/except 包住了所有 requests 调用。如果捕获到 `ConnectionError`，说明后端没起来，就在页面上显示一个红色错误提示"无法连接到后端，请确认 uvicorn 已启动"，而不是崩溃。这种容错处理在正式项目里是标配——不管后端挂了还是网络断了，前端都要优雅降级。                                                                                                                               |
| ⭐   | **upload 时为什么传的是文件内容而不是文件路径？**                 | 因为 Streamlit 的文件上传组件返回的是内存里的字节流——用户点了文件之后文件内容就在 Streamlit 进程的内存里了，不是一个磁盘路径。所以我用 `file.getvalue()` 拿到字节流，和文件名一起封装成 `multipart/form-data`，通过 HTTP POST 发给 FastAPI。FastAPI 那边收到后再把字节流存成临时文件给 PyPDFLoader 用。整个过程不让用户操心路径问题。                                                                         |
| ⭐   | **st.file_uploader 和 st.chat_input 分别是做什么的？**  | `st.file_uploader` 是文件上传按钮，type=["pdf"] 限制只选 PDF，选了之后返回文件的字节流和文件名。`st.chat_input` 是类似 ChatGPT 的那个对话框输入栏，用户打完字回车，它返回输入的文本。这两个是 Streamlit 最核心的交互组件，分别对应 /upload 和 /chat 两个接口。                                                                                                                     |

## ✅ Day 6：完善 UI

| 优先级 | 问题                                              | 口语回答                                                                                                                                                                                                |
| --- | ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐  | **你的项目做了哪些 UI 优化？**                             | 核心是四个改进。第一，上传后状态持久化——用 session_state 存文件名和统计信息，刷新页面不丢。第二，清空对话按钮——一键清除历史记录。第三，来源展示美化——改成 HTML 卡片样式，比纯文本更清晰，面试官一眼能看到来源页码。第四，上传前禁用输入框——没上传 PDF 时输入框灰色显示"请先上传 PDF"，防止报了空提问的错误。                          |
| ⭐⭐  | **为什么要加 GET /status 接口？**                       | 因为前端需要知道后端当前的状态——有没有 PDF 已索引、文件叫什么、切了多少块。没有这个接口，前端只能在提问失败时靠 try/except 推断"哦，可能没上传文件"。加了 /status 之后，页面一加载就去查状态，如果后端有文件就直接恢复显示，如果后端没启动也能提前告知用户，而不是等到提问才报错。这也符合 RESTful 的设计原则——资源状态查询就该用 GET。          |
| ⭐⭐  | **Streamlit 的 st.rerun() 是什么？为什么 upload 后要调它？** | st.rerun() 让 Streamlit 立即重新执行整个脚本。我上传 PDF 成功后把文件信息存进了 session_state，但如果不 rerun，页面上显示的还是旧的"尚未上传 PDF"的状态。调了 rerun 后页面从头跑一次，session_state 里已经有新数据了，侧边栏立刻显示文件信息卡片。这是 Streamlit 的标准模式——改状态 → rerun 刷新视图。 |
| ⭐⭐⭐ | **前后端分离架构中，前端怎么处理后端不可用的情况？**                    | 分了三种情况处理。第一是启动时——页面加载就调 GET /status，如果连不上，侧边栏显示🔴"后端未连接"，并提示用户启动 uvicorn。第二是上传时——如果 ConnectionError，显示红色错误提示。第三是提问时——同理处理。前端不因为后端挂了就崩溃，而是显示友好的错误信息，这叫优雅降级。                                          |
| ⭐   | **st.chat_input(disabled=True) 是什么？**           | 条件禁用输入框。当 `has_doc=False` 时 disabled=True，输入框变成灰色不可写，placeholder 变成"请先在左侧上传 PDF 文件"。用户传了 PDF 后 disabled 变成 False，恢复正常输入。这个小细节能防止用户在不该提问的时候提问，减少错误提示的干扰。                                             |
| ⭐   | **来源展示为什么用 HTML 而不用 st.markdown？**              | 因为我想控制卡片样式——边框、圆角、背景色、内边距。st.markdown 只能渲染纯 Markdown 文本，做不了布局。所以我用 unsafe_allow_html=True 传一段带 style 属性的 HTML 片段，把每个来源包成一个圆角卡片，上面灰字显示"来源 X · 第 N 页"，下面正文显示内容摘录。这样比纯文本清爽很多，面试官打开来源看到也舒服。             |

## ✅ Day 7：Reranker（BGE Cross-Encoder 精排）

| 优先级 | 问题                                     | 口语回答                                                                                                                                                                                                                                                                                                      |
| --- | -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ⭐⭐⭐ | **Reranker 是什么？为什么需要它？**               | 向量搜索其实挺糙的。比如你搜"RAG 的三大优势"，跟 RAG 有关的文档距离都近——有讲优势的，也有讲组件的，还有讲高级技术的，它们跟问题的语义距离差不多。向量搜索只能把它们都排前面，不分好坏。Reranker 就是干精排的活：先用 embedding 粗检索 8 块回来，然后用 Cross-Encoder 模型对这 8 块逐条打分，看每块跟问题的相关度到底有多高，最后只取分数最高的 3 块给 LLM。它的好处是精度高，缺点是比向量搜索慢一点，所以"先粗排后精排"是标准做法。                                                         |
| ⭐⭐⭐ | **Bi-Encoder 和 Cross-Encoder 的区别？**    | Bi-Encoder（双向编码器）就是 embedding 模型——它把问题和文档分别编码成向量，然后算余弦相似度。好处是文档向量可以预先算好存起来，检索时只编码问题就行，很快。坏处是问题和文档在编码时没有交互——各自独立编码，靠向量距离判断关系，不够精准。Cross-Encoder（交叉编码器）不一样——它把问题和文档拼接在一起输入，让模型同时看到两者，然后输出一个 0 到 1 的相关性分数。好处是准，因为它能看到问题和文档之间的交叉关系。坏处是慢——每对都要重新跑一次模型，不能预计算。所以 RAG 里是"Bi-Encoder 粗召回 + Cross-Encoder 精排"，各取所长。 |
| ⭐⭐  | **你的 Reranker 用的是什么模型？怎么嵌入到 RAG 链路的？** | 我用的是 BAAI 的 bge-reranker-base，基于 BERT 的 Cross-Encoder 模型。嵌入方式：我没有把它塞进 LangChain 的链里面，而是在 ask() 方法里手动做三步——①用向量检索器搜 8 块回来；②把每块和问题拼成 (question, doc) 对，用 Cross-Encoder 逐对打分；③按分数降序排序，取前 3 块拼成 context 给 LLM。为什么手动做而不是用 LangChain 的 CompressionRetriever？因为手动做代码更透明，面试可以把每一步讲清楚，而且不受 LangChain API 版本变动影响。    |
| ⭐⭐  | **加了 Reranker 之后检索流程跟之前有什么不同？**        | 之前（Day 6）是向量检索直接取 top 3 → 给 LLM。现在（Day 7）是向量检索取 top 8 → Reranker 打分排序 → 取 top 3 → 给 LLM。k 从 3 变成 8（粗召回），但最终给 LLM 的还是 3 块，只是这 3 块经过了二次筛选，质量更高。代价是每次提问多跑一次 Cross-Encoder 推理（8 对打分），大概多花 0.5 到 1 秒。                                                                                                          |
| ⭐⭐  | **Reranker 模型下载要多久？懒加载是什么意思？**         | bge-reranker-base 大约 300MB，下载一次之后缓存到 huggingface 的模型目录。我在引擎里用了懒加载——不是启动时就下载，而是第一次调用 upload 时才触发下载。这样如果只是启动后端查 /status，不会浪费时间和带宽。面试官可能会关心产品级别的启动速度，懒加载是常见的优化手段。                                                                                                                                            |
| ⭐   | **为什么用 base 而不用 large 版本的 Reranker？**  | bge-reranker-large 确实效果更好，但模型大一倍（560MB），推理更慢。我用 base 版本在 CPU 上每对打分大概几十毫秒，8 对就是 0.5 秒以内，加上向量检索和 LLM 生成，整个问答流程控制在 3 秒左右。如果用 large，可能会涨到 5-6 秒，用户体验就差了。而且对于 PDF 知识库这种场景，base 的精度已经够了——毕竟上下文很短，不需要特别强的语义理解。                                                                                                   |
| ⭐   | **Reranker 的 predict() 返回什么？你怎么用它的？**  | `cross_encoder.predict([("问题", "文档1"), ("问题", "文档2"), ...])` 返回一组浮点数，每个数是该对的相关性分数（越高越相关）。我拿到这些分数后，和文档列表一起 zip，按分数降序排序，取前 self.final_k 个文档。这样就完成了"向量粗排 8 → Cross-Encoder 精排 3"的流程。                                                                                             |
